Le Pinning — le piège à ne pas rater en entretien

Le pinning arrive quand un Virtual Thread reste cloué à son carrier pendant un blocage. Le carrier ne peut plus servir d'autres VTs.

Cause : synchronized en Java 21–23. Solution : ReentrantLock. Résolu nativement en Java 24 (JEP 491).

Au supermarché

Il y a {{ nCaisses }} caisses ouvertes et {{ nClients }} clients à servir. Chaque client arrive à la caisse — et là, il réalise qu'il doit trouver sa carte bleue au fond de son sac. Ça prend du temps.

La question, c'est : que fait le caissier pendant ce temps ?

synchronized Le caissier reste planté. Il attend. Il bloque la caisse entière. Quand tous les caissiers attendent en même temps → le supermarché freeze.
ReentrantLock Le caissier lance l'attente, puis se libère presque aussitôt. Quand la carte est prête, il revient finir l'encaissement. Throughput maintenu.

En Java : le "client qui cherche sa carte" = l'I/O (appel BDD, requête HTTP). Le "caissier" = le carrier OS. Lance la simulation.

Ta lecture est la bonne

Avec ReentrantLock, le carrier ressemble à un caissier mobile : il passe rapidement d'une caisse à l'autre pour démarrer ou terminer une opération, puis laisse le client attendre sans immobiliser la caisse.

Avec synchronized en Java 21, le carrier reste attaché à la caisse pendant toute l'attente. Il ne peut pas aller aider ailleurs, même si le virtual thread ne fait rien de CPU à ce moment-là.

Le point important

ReentrantLock ne rend pas la section critique parallèle : un lock partagé reste un passage un par un. La différence, c'est ce qui arrive pendant l'attente. Avec synchronized, l'attente garde un carrier. Avec ReentrantLock, l'attente parque le virtual thread et rend le carrier au scheduler.

Sans lock
~30ms
200 VTs en parallèle — aucune attente
synchronized
~6000ms
Séquentiel + carrier bloqué → freeze sous charge
ReentrantLock
~6000ms
Séquentiel aussi — mais carrier libéré

Pourquoi les temps sont identiques ? Un seul lock partagé = une seule tâche à la fois = séquentiel dans les deux cas.

La vraie différence se voit sous charge. synchronized sature les carriers OS → l'appli freeze. ReentrantLock libère les carriers → les autres VTs continuent.
